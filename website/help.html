Run Config reference

Mosaic groups its knobs into four popups in the Run Config panel.

ReCom

Number of Districts
How many districts to draw. Set to the chamber size (e.g. 14 for NC's US House delegation, larger for state legislatures). Mosaic will refuse to start if the precinct count is too small to support the requested number of districts.
Max Iterations
Hard cap on the optimisation budget. The annealing schedule is sized against this number, so changing it changes the cooling trajectory, not just the stopping point.
n=3 ReCom Mix
Probability per step of attempting a three-way recombination (merge 3 adjacent districts and re-split into 3) instead of the default two-way step. Three-way moves are slower but escape local minima better; 0.0 disables it.
Polish Flips (on by default)
Adds single-precinct boundary flips alongside ReCom to clean up district borders late in the run. The flip probability ramps along a two-piece logistic: about 5% at the start, rising to 50% at the crossover, and 85% by the end — so at least 15% of steps stay ReCom throughout and the chain can still escape local minima. The 50% crossover slider (default 84% of the run) sets where the rate hits 50%; higher keeps flips rare until later. The live Flip rate readout in the status panel shows the current ramp value during a run.

Population

Population Tolerance
Per-district deviation from the ideal population that Mosaic will accept as a valid plan. Any candidate that violates this is rejected at the recombination step (default 2.5%). Tighter is harder to satisfy and slower to converge.
Tolerance Ratchet (Off / Standard / Strict)
Progressively tightens Population Tolerance over the back two-thirds of the run, from the value you set down toward 0.25%. The schedule only ever tightens, and never below the deviation the map has actually achieved — so it can't strand a plan it is unable to satisfy. Standard tightens only when a new best plan is found; Strict tightens at every eligible iteration; Off (default) leaves the tolerance fixed. Scoring always uses your fixed map-wide tolerance — the ratchet acts on the move-acceptance bound, squeezing population balance down as the run settles. While on, the Population Deviation score is tracked every iteration even at weight 0 — its score-row label turns green to signal the ratchet is driving it, and its chart records data regardless of weight.
Safe Harbor
The no-penalty band for the Population Deviation score (default 0.0%, i.e. off — every district outside perfect balance is penalized). Districts inside the band contribute 0 to that metric; districts outside contribute their squared excess deviation. Clamped at run-start to never exceed Population Tolerance.

Annealing

GUIDED cooling (recommended)
Temperature drops fastest during an initial guide window (a fraction of max iterations) and tapers off afterward. The schedule self-tunes to acceptance rate, so the same settings work across very different shapefiles and weight mixes.
STATIC cooling
Temperature decays on a fixed geometric schedule. Useful when reproducing a published methodology or when you want fine control over the cooling curve. Requires more manual tuning to match GUIDED's quality.
Initial Temp Factor
The GUI exposes the factor slider (default 0.2). In the default PROPORTIONAL mode the starting temperature is factor × initial_score — the right behavior when you don't know what magnitudes to expect from your weight mix, since temperature scales with whatever the score actually is. (A NOMINAL mode that treats the factor as an absolute temperature exists for headless/config use but is not surfaced as a GUI control.)
GUIDED — Guide Point & Target Temperature
The schedule is sized so that temperature reaches Target Temperature at Guide Point × max_iterations. Defaults: 0.9 and 1.0. Lowering Guide Point means faster cooling (more aggressive); lowering Target Temperature means a colder endpoint (less random late in the run). (The underlying config field is guide_fraction.)
STATIC — Cooling Rate
Per-iteration temperature multiplier (default 0.9995, meaning temp drops 0.05% per step). The total cooling over a run is cooling_rate^max_iterations; e.g. 0.9995 over 10,000 iterations leaves temperature at about 0.7% of initial.
Launch Watch
When enabled (default on), Mosaic re-anchors the temperature to the current score after a set number of iterations, configurable via the Re-anchor after iter input (default 250). Useful when score collapses early (common with County Splits and County-Edge Bias enabled) and the initial-anchored temperature would otherwise become too hot to be useful by the time the score has stabilized.
n=3 ReCom Mix slider
Frozen at run start (see ReCom). Listed here too because the Annealing popup is where it lives in the GUI.

Seed

Random Seed
Sets the RNG state at the start of the run. 0 (default) leaves the RNG random, so every run is different. Any non-zero integer makes a run reproducible on the same machine, with the same Mosaic version, with the same shapefile. Cross-machine or cross-version runs may diverge slightly due to floating-point ordering inside numpy / igraph; the seed buys you reproducibility, not bit-exactness across environments.
Hot start (Advanced menu)
Instead of beginning from a random seed plan, load an existing precinct-to-district assignment as the starting point. Pick a CSV with an id column and a district column (1-indexed; the format matches Mosaic's own assignment export), then map the id column to your shapefile. Mosaic validates that every precinct is assigned, the district count matches Number of Districts, and each district is contiguous before the run will use it. Useful for polishing a published map or resuming from a prior result. Clear it from the same menu to return to random seeding.
Relight (Advanced menu)
A "keep refining" mode. With a paused or finished run on screen, arming Relight makes each subsequent Start reseed from the current map instead of a random plan — so runs chain, the result of one becoming the seed of the next (unlike Hot start, which pins a fixed CSV). It also applies a gentle polish preset to the Annealing settings — a very low initial temperature, a long guided-cooling window, a heavy n=3 ReCom mix, late-onset flips, and Launch Watch off — so a run settles the existing map rather than tearing it up. It leaves your weights, Population Tolerance, and everything else alone. Mutually exclusive with Hot start (each greys the other out). Clear Relight turns it off and restores the annealing settings to exactly what they were when you armed it; loading a new shapefile clears it automatically. A run is refused if the current map no longer matches the Districts count or the Population Tolerance.

County-Edge Bias

County-Edge Bias (enable + Multiplier slider)
Makes ReCom less likely to propose cut edges that cross county lines when generating new district boundaries. Higher multiplier = stronger preference for cuts along county borders. Distinct from the County Splits score: County Splits penalizes splits in the score function (post hoc), County-Edge Bias steers the proposal distribution (per step). The two work well together — County-Edge Bias keeps splits rare in proposals; County Splits cleans up the ones that do happen.

Scoring weights

Each enabled metric is multiplied by its weight and summed into PlanScore.total, which the optimizer minimizes. Weight 0 disables the metric; the computation is skipped, reducing per-iteration cost. Per-metric contribution formulas are in the Metrics glossary.

Partisan calibration

Set in the Partisanship popup. Applied when any of Efficiency Gap (Robust), Expected Dem Seats, Competitiveness, Chance of Majority, or Hinge is enabled.

election_win_prob_at_55 — default 0.9
Calibration anchor: P(D wins | D share = 0.55). Inverted to σ_district = 0.05 / Φ⁻¹(p). Higher values yield sharper per-district win probabilities.
election_swing_sigma — default 0.03 (3pp)
Standard deviation of a shared partisan-environment swing added to every district before computing win probability. Higher values widen the partisan metric distributions.

Combined as σ_combined = √(σ_swing² + σ_district²). For point-estimate behavior, set election_swing_sigma to its minimum and disable Robust EG (Partisanship popup → Efficiency Gap mode → Static).

Run controls

Four buttons in the main panel drive a run.

Start
Begins (or resumes) optimisation with the current Run Config and score weights. Most settings are frozen at run start — including the n=3 mix probability and the annealing schedule — so changes to the popups during a run don't take effect until you Reset.
Pause
Stops the algorithm thread at the next clean iteration boundary. Charts and the map stay where they were. Click Start again to resume from the same iteration.
Revert to Best
Restores the current assignment to the best-scoring plan seen so far in this run, without resetting the iteration counter or temperature. Useful when the algorithm has wandered into a worse region and you want to continue annealing from a known-good point. The score history charts keep their full trace. The button greys out when the map already is the best plan, since there is nothing to revert to.
Reset
Discards the current run entirely. Clears the map, the iteration counter, the score history, and the annealing state. The loaded shapefile and column choices are kept. Use this when you want to try a fresh run with new weights or a new seed.

Renumbering districts

By default Mosaic numbers districts in the arbitrary order it draws them. The Advanced menu can renumber a finished map by geography — numbers only. Colors stay put, so a district keeps its color and simply shows a different label on the map, in the District Info panel, and in exported CSVs. Renumbering is deterministic and never touches the assignment or any score.

Renumber districts after run (on by default)
When a run finishes, relabel automatically using the rule chosen in Renumber options. Turn it off to keep the draw-order numbering.
Renumber options
Pick the rule:
  • None — draw-order numbering (no renumber).
  • NW to SE — diagonal sweep: district 1 sits furthest northwest, the last furthest southeast. Each axis is normalized so the diagonal stays balanced regardless of the state's shape.
  • North to South — district 1 is northernmost, ties broken west-to-east.
  • Infer from alignment — available only with a reference plan loaded (see the Alignment metric). Each proposed district adopts the number of the reference district it most overlaps, decided by a global maximum-overlap matching so two districts never claim the same number; any surplus districts get fresh numbers. Lets a new map carry the enacted map's district numbers.

Metrics glossary

Each metric appears on a score row in the GUI. Entries below give a definition followed by a meta card listing the unit, optimization direction, contribution to PlanScore.total, and any tunable parameters with their GUI location. Metrics are internally rescaled to comparable magnitudes; weights and formulas appear in the meta card.

Five metrics (Efficiency Gap in Robust mode, Expected Dem Seats, Competitiveness, Chance of Majority, Hinge) model each district's Dem share as a Gaussian variable. Two parameters set the spread and are shared across these metrics: election_win_prob_at_55 (anchors per-district noise via σ_district = 0.05 / Φ⁻¹(p)) and election_swing_sigma (standard deviation of a shared partisan-environment draw). They combine as σ_combined = √(σ_swing² + σ_district²). Both are set in the Partisanship popup; see Run Config > Partisan calibration for details.

Cut Edges

Number of precinct-pair adjacencies whose endpoints are in different districts.

County Splits

Penalty on splitting counties across districts. Two terms: excess splits beyond the minimum forced by county population, plus the shortfall in single-county districts (districts whose precincts all lie within one county). In the GUI this score shares one row labeled County Splits and Bias, which also houses the County-Edge Bias toggle (a proposal-time steer, distinct from this post-hoc penalty).

Unitweighted score
Directionlower is better
Contribution10 × (excess_splits + (max_single_county − single_county_districts))
Tunablesnone directly. max_clean depends on the run-config Population Tolerance (Population popup → Population Tolerance).
Population Deviation

Sum-of-squares imbalance penalty. Districts within the safe-harbor band contribute zero. Beyond it, the metric sums each district's squared excess deviation, normalized by district count — a smoother optimization landscape than the older mean-plus-max form.

Unitscaled sum of squares
Directionlower is better; zero indicates all districts inside the safe-harbor
Contributionsum(excess²) × 100,000 / n_districts, where excess = max(0, |dev_frac| − safe_harbor)
TunablesPopulation popup → Safe Harbor slider (pop_deviation_safe_harbor, default 0.0%). Clamped at run start to not exceed Population Tolerance.
Alignment

Least-change penalty: how close the plan stays to a loaded reference plan (e.g. the enacted map). District numbering is irrelevant — only the partition matters. Each reference district is scored by its cohesion: the probability that two of its residents are still in the same proposed district (the sum of its squared overlap fractions). A district kept whole costs nothing; one split apart costs more, and a clean two-way split costs less than a shatter. The score is one-sided by design — it penalizes breaking old districts apart, not merging them — so population balance and the other weighted terms are what rule out the degenerate "everything in one district" plan. Load a reference CSV from the Alignment row → "..." settings before enabling.

Unitcontribution [0, 100]; chart shows cohesion % in [0, 100]
Directionlower contribution is better; 0 = identical to the reference plan
Contribution100 × Σ wₐ(1 − cohesionₐ) / Σ wₐ, where cohesionₐ = Σₚ fₐₚ² over reference district a's share f in each proposed district p, weighted by w (population, or a party's votes)
Tunables
  • Alignment row → "..." → Load reference plan (CSV with an id column and a district column; the district count need not match the run, and the reference need not be contiguous or population-balanced)
  • Alignment row → "..." → Focus (alignment_party_focus): measure retention in all residents (population), or in a party's votes — so each district scores on how many of its partisans stayed together. Needs election data; falls back to population without it
  • Alignment row → "..." → Only districts that party wins + win threshold (alignment_restrict_to_party, alignment_win_threshold, default 0.535): score only the reference districts the focus party already holds, to preserve your own seats. Needs a party focus and election data
Compactness

Mosaic's primary compactness score — on by default at weight 50. One dial that blends mean Polsby-Popper and mean Reock 50/50, each linearly normalized into [0, 100] over a fixed band, so you can optimize compactness without juggling the two separate weights.

Unitcontribution [0, 100]
Directionlower contribution is better
Contribution100 − rating
Polsby-Popper

Compactness penalty. Per-district PP = 4π·area / perimeter² ranges from 0 (elongated) to 1 (circle). The score chart displays mean(PP) directly; the optimizer minimizes (1 − mean(PP)) × 100. Off by default — the blended Compactness score (above) is Mosaic's primary compactness metric; enable Polsby-Popper on its own only when you want to tune it directly.

Unitcontribution [0, 100]; chart shows mean PP in [0, 1]
Directionlower contribution is better
Contribution(1 − mean(PP)) × 100
Reock compactness

Second compactness penalty, complementary to Polsby-Popper. Per-district Reock = area / area(minimum bounding circle) ranges from 0 (elongated) to 1 (circular). Polsby-Popper measures boundary smoothness; Reock measures how round the district is overall. A district can be smooth-edged but stretched thin (high PP, low Reock), or jagged but compact in extent (low PP, high Reock). Optimizing both together usually catches what either misses alone.

Mosaic uses a 16-direction approximation rather than the textbook minimum bounding circle, which is too slow to evaluate on every iteration. The approximation caches each precinct's 16 directional extreme vertices once at load, then computes the bounding diameter from those candidate points. Agreement with textbook Reock is within ~0.8 score points on a 0–100 penalty scale. The optimizer treats the approximation as a canonical deterministic score, identical to PP or any other component.

Unitcontribution [0, 100]; chart shows mean Reock in [0, 1]
Directionlower contribution is better
Contribution(1 − mean(Reock)) × 100

mean(district Dem share) − median(district Dem share). Non-zero values indicate the mean and median districts disagree on partisan balance. Sign: negative = D advantage, positive = R advantage.

Tunables

Map overlays

Checkboxes under the live map switch what the map shows. Only one of the colouring modes can be active at a time; the others (county lines, labels) stack on top.

County
Draws grey lines along county boundaries on top of the current colouring. Useful for eyeballing how often district boundaries follow county lines.
Splits
Dims counties that lie entirely inside a single district, leaving split counties at full saturation. Quick visual diagnostic for the County Splits score: more colour = more splits.
Pct. Results (Precinct Results)
Colours each precinct by its Dem vs Rep share (Classic Mosaic red ↔ blue palette). Independent of district boundaries; shows the partisan terrain Mosaic is working over. Requires election columns at load.
Dist. Results (District Results)
Colours each district by its aggregate Dem share. The map view of the same data the Mean-Median, Efficiency Gap, and Competitiveness metrics summarise.
Compactness
Colours each district by its combined compactness — the same Polsby-Popper + Reock blend the Compactness score uses (round = greener, elongated = redder). Spot the ugly district at a glance.
Pop. Dev. (Population Deviation)
Colours each district by signed deviation from ideal population. Underpopulated districts in one direction, overpopulated in the other.
Labels
Draws the district number at each district's interior. Number matches the labels in the District Info panel and the exported CSVs.
Precincts
Draws faint white precinct boundaries on top of the district colouring. Hairline at on-screen resolution; cleans up at high-DPI exports. Useful for previewing how district lines fall relative to precinct lines.

Reading the charts

Two menus control what you see, and they are easy to confuse: the Scores menu toggles which score rows appear in the left-hand config column, while the Views menu toggles the right-side charts and panels below. The same names (e.g. Mean-Median, Reock) appear in both menus for that reason. The most useful panels during a run:

Score History
Total PlanScore.total over iterations (lower is better). Trends down as annealing cools and the search becomes pickier.
Per-metric panels
One chart per enabled metric, plotted in its native units. Partisan metrics draw a reference line where relevant.
Alignment
When the Alignment metric is active, this panel plots cohesion — the share of each reference district that stays intact — as a mean across districts and the single worst-hit district. The axis is inverted relative to the score: higher here means closer to the reference, while the score row reports the penalty (lower is better).
Score Contributors
Bar chart showing the share of the total score each enabled metric contributes right now. The quickest way to see which weight is actually driving the optimizer.
District Info
Per-district table: population, deviation from ideal, and (when election data is loaded) Dem share. District numbers match the map labels and the exported CSVs. Refreshes on an adjustable interval to stay cheap on long runs.
Win Chance / Partisanship
Partisan diagnostic panels: per-district win probabilities and the partisan distribution of the current plan. Available once election columns are loaded.
Temperature
The annealing temperature over time. In GUIDED mode the curve is non-linear by design; in STATIC mode it's a clean geometric decay.
Entropy / Acceptance
Fraction of recent proposals that were accepted. Falls as the temperature drops; useful diagnostic for whether the schedule is too aggressive (drops to near-zero too early) or too gentle (stays high throughout).

Most charts will downsample the displayed buffer once it exceeds about 10,000 points so the GUI stays responsive across long runs. The underlying data isn't lost — it just isn't all drawn.

Exporting results

Mosaic writes two CSVs from the toolbar buttons in the main panel. Both save the plan currently on the map — exactly what you see on screen. Controls like Revert to best change what's on the map; the export just follows.

Save Assignments → output/assignments_YYYYMMDD_HHMMSS.csv
One row per precinct: the precinct's ID (from whichever column you picked at load) and the district number. District numbers match the labels shown on the live map — same colours, same labels, no surprises.
Save Metrics → output/metrics_YYYYMMDD_HHMMSS.csv
One row per district: population, deviation from ideal, optional vote totals and Dem/Rep percentages, optional Polsby-Popper compactness. Same district numbering as the assignments CSV.

Both files load directly into Dave's Redistricting App via its Color Map from File feature, or join back to your shapefile in QGIS / ArcGIS using the precinct ID column.

Map image (PNG)

Two icons sit at the right end of the map-overlay row, just under the District Map panel.

Camera icon → output/map_YYYYMMDD_HHMMSS.png
Quick PNG of the current map. Re-rasterized at native resolution and cropped to the state's bounding box (so vertical states like Mississippi save tall, not letterboxed). Preserves whichever overlays are currently checked.
"..." (Photo Menu) icon
Opens an export dialog for high-resolution output: optional title bar, DPI selection (96 / 144 / 192 / 288 / 384 / 576 — 288 is the default), and a "Made with Mosaic" caption strip across the bottom. The map is re-rasterized at the chosen resolution (not upscaled), border and label thicknesses scale at about half rate so high-DPI exports don't over-emphasize the lines, and a black state-edge outline is added so the boundary reads cleanly when the image is shared.

FAQ & troubleshooting

Mosaic crashed during a run. Where's the log?
Top-level crashes are written to crashes/YYYYMMDD-HHMMSS.log next to where you launched from. The file includes the traceback, Mosaic version, platform, and any context the runner had (current iteration, shapefile path). Attach it when filing an issue.
Mosaic won't load my shapefile.
See the shapefile guide — in particular the "If Mosaic refuses to load" section, which maps each error message to the fix in the source data.
My state has islands (HI, AK, the Outer Banks…). Do I need to remove them?
No. When Mosaic builds the adjacency graph it automatically reconnects islands and exclaves to the mainland with virtual bridge edges, so ReCom can reach every precinct. The bridges are chosen deterministically by geography (nearest precinct, same county preferred) and are invisible to scoring — they never count as cut edges and the file on disk is never changed. The "disconnected components" error now appears only in the rare case where bridging can't find a valid link (e.g. degenerate geometry).
The launcher fails on first run.
See the install page troubleshooting section — SmartScreen on Windows, Gatekeeper on Mac, and a few common uv install failure modes.
Why is the first iteration slow?
Numba JIT-compiles the hot ReCom path on the first ReCom step (several seconds). Subsequent steps run at hundreds per second. The JIT cache persists across runs, so this only hurts once.
Can I run multiple chains in parallel?
Not yet from inside Mosaic. As a workaround, run multiple Mosaic instances in separate folders with different seeds.
Mosaic feels slow / the GUI is laggy.
Two knobs in the Appearance menu help. Limit plots trims the score-history buffer to keep chart rendering fast on long runs (the underlying score data isn't lost, just downsampled for display). Map render interval sets how often the map redraws during a run; raise it from the default to give the algorithm more CPU. The algorithm itself runs at the same speed either way; both knobs only affect rendering.

About & methodology

Mosaic is a Python port of Mosaic for R, extended with a live Dear PyGui interface and a simulated-annealing layer on top of the underlying ReCom step.

The ReCom (recombination) algorithm itself is derived from the one developed by the MGGG Redistricting Lab. At each step, Mosaic picks an edge that crosses two districts, merges those districts, draws a random spanning tree on the merged region, and cuts the tree to produce two new districts of balanced population. The simulated-annealing wrapper accepts or rejects each proposed step based on a weighted plan score plus a temperature schedule.

Relationship to canonical ReCom, MergeSplit, and GerryChain. Mosaic borrows the recombination proposal from ReCom but accepts each proposal under simulated annealing keyed to a weighted plan score, rather than under a Metropolis-Hastings rule designed to produce a known stationary distribution over valid partitions. That distinction matters: canonical ensemble samplers — ReCom in GerryChain and the corrected MergeSplit variant in redist — are designed so that the distribution of accepted plans supports statistical inference (for example, "this enacted plan is an outlier compared to the distribution of valid plans"). Mosaic, with the simulated-annealing layer biasing the chain toward score improvements as it cools, instead produces an optimized plan under the user's weighted objectives. Mosaic runs should not be cited for ensemble-style outlier claims; use redist or GerryChain for that.

Islands and connectivity. ReCom needs a single connected adjacency graph. When a shapefile has islands or exclaves, Mosaic links each detached component to the mainland with one virtual bridge edge, chosen deterministically from geometry (nearest precinct, same county preferred). Virtual edges keep the graph connected so moves can reach an island, but they are excluded from the cut-edge count and every other score, and the shapefile on disk is never modified.

Some features from the R version are implemented differently here, and some are still in progress.
